# Automatic Priority Adjustment Design

## Goal

Automatically maintain channel priority by combining nominal price, cache benefit, recent availability, first-token latency, and throughput.

This feature is the next scheduling step for Wynth API after strict priority fallback and upstream source synchronization. It must not change request-time channel selection semantics. The scheduler should continue to select by the existing `priority` and `weight` fields; this feature periodically updates `priority` for managed channels.

## Confirmed Decisions

- Only channels generated by upstream sources are managed.
- Manual channels are never changed by automatic priority adjustment.
- The feature is opt-in. Existing upstream sources and generated channels keep their current priority until automatic priority is enabled.
- Cache benefit is an independent score signal. It must never make an expensive channel nominally cheaper or alter the 8x nominal-price invariant.
- The first implementation updates the existing `channels.priority` field instead of adding a parallel request-time score.
- Channel `weight` remains unchanged in this step.

## Non-Goals

- No account-pool scheduler work.
- No request-time dynamic scoring.
- No change to strict priority retry behavior.
- No automatic weight adjustment.
- No availability/cache/latency dashboard beyond showing the latest priority score explanation.
- No global automatic priority switch for every channel.

## Current Data Sources

### Upstream Price

Upstream source discovery stores group pricing in `model.UpstreamSourceChannelMapping`:

- `EffectiveRateMultiplier`
- `LocalChannelID`
- upstream group name, platform, status, and discovery status

Generated channels also store ownership metadata in `dto.ChannelOtherSettings`:

- `generated_by_upstream_source_id`
- `generated_by_upstream_mapping_id`

The source of truth for the price component is the mapping's `EffectiveRateMultiplier`, not the channel name or remark.

### Availability

Channel monitor history is stored in `model.ChannelMonitorLog`.

Useful fields:

- `status`
- `checked_at`
- `latency_ms`
- `first_token_latency_ms`

`model.GetChannelMonitorStats(channelIDs, windowStart)` already returns total checks, successful checks, and average latency values for a rolling window.

### Real Usage Cost Signals

Real request logs are stored in `model.Log` with:

- `type = model.LogTypeConsume`
- `channel_id`
- `created_at`
- `prompt_tokens`
- `completion_tokens`
- `other`

The `other` JSON contains billing and performance fields such as:

- `completion_ratio`
- `cache_tokens`
- `cache_ratio`
- `cache_creation_tokens`
- `cache_creation_ratio`
- `cache_creation_tokens_5m`
- `cache_creation_ratio_5m`
- `cache_creation_tokens_1h`
- `cache_creation_ratio_1h`
- `cache_write_tokens`
- `input_tokens_total`
- `frt`

Automatic monitor probes do not write consume logs. Manual channel tests can write consume logs, so the usage stats reader must exclude test logs. New test logs should include an explicit marker such as `other.is_channel_test = true`; legacy test logs should also be excluded when `content` or `token_name` is `模型测试`.

## Target Behavior

When automatic priority is enabled for an upstream source or a matched source rule:

1. The background worker finds generated channels whose resolved automatic-priority schedule is due.
2. It loads their upstream mappings and current channels.
3. It computes a nominal price score from the source/channel rate multiplier only.
4. It computes an independent cache score from the guarded cache-cost factor.
5. It computes availability, first-token latency, and throughput scores.
6. It combines those scores using the 75% / 10% / 8% / 3% / 4% weights.
7. It maps the final score to a `priority` value.
8. It writes the new priority only when the change passes a hysteresis threshold.
9. It updates the corresponding `abilities.priority` values and refreshes channel cache state so request-time scheduling sees the change.
10. It stores the latest score breakdown for explainability.

The worker must never enable, disable, delete, or create channels. It only adjusts priority and score metadata for channels it owns.

## Configuration Model

Extend upstream source sync config with source-level defaults:

```json
{
  "auto_priority_enabled": false,
  "auto_priority_interval_minutes": 30,
  "auto_priority_window_hours": 24
}
```

Extend local group rules with an optional override:

```json
{
  "auto_priority": {
    "enabled": true,
    "interval_minutes": 15,
    "window_hours": 48
  }
}
```

Resolution rules mirror monitor and auto-sync settings:

- If a rule matches and sets `auto_priority.enabled`, it overrides the source setting.
- If a rule matches and sets interval/window values, those override source values.
- If a rule omits a field, the source-level value is used.
- If no rule matches, the source-level fallback is used.
- Disabled rules or unsynced mappings are not eligible.

Defaults:

- automatic priority: disabled
- interval: 30 minutes
- metrics window: 24 hours
- maximum metrics window: 168 hours
- interval `0`: due on every worker tick, intended for debugging/manual local use
- positive intervals are preserved verbatim; only negative intervals normalize to the 30 minute default
- negative windows normalize to defaults

## Score Model

### Final Score

The final score is intentionally price-dominant:

```text
final_score = effective_price_score * 0.80
            + availability_score     * 0.17
            + first_token_score      * 0.03
```

All component scores are normalized to `0..100`.

The output priority is:

```text
priority = round(final_score * 10)
```

This gives a default managed range of `0..1000`, which is large enough for strict priority ordering while still being easy to inspect.

### Effective Price

Effective price is the upstream multiplier corrected by observed cache economics:

```text
effective_cost_multiplier = effective_rate_multiplier * cache_adjusted_cost_factor
```

`effective_rate_multiplier` comes from `UpstreamSourceChannelMapping.EffectiveRateMultiplier`.

`cache_adjusted_cost_factor` is derived from recent real usage logs. It represents how expensive the channel's recent workload was compared with the same workload without cache discounts.

For each real request:

```text
normal_units =
  total_input_tokens_at_normal_price
  + completion_tokens * completion_ratio

adjusted_units =
  uncached_input_tokens
  + cache_read_tokens * cache_read_ratio
  + cache_write_tokens * cache_write_ratio
  + completion_tokens * completion_ratio

request_cache_factor = adjusted_units / normal_units
```

Aggregation is weighted by `normal_units` so tiny requests do not dominate the estimate:

```text
cache_adjusted_cost_factor =
  sum(adjusted_units) / sum(normal_units)
```

Parsing rules:

- OpenAI-style cached reads use `cache_tokens` and `cache_ratio`.
- Claude-style cache creation/read uses the cache creation fields when present.
- If split 5m/1h cache creation fields are present, use the split fields and do not double-count the aggregate `cache_creation_tokens`.
- Cache creation ratios are allowed to be greater than `1`. In that case cache creation raises the effective cost for the sampled workload instead of lowering it.
- `input_tokens_total` is preferred when present because it is the normalized total input token count already recorded by billing.
- If a request has no usable token denominator, ignore that request for cache adjustment.
- If no real usage logs remain, use `cache_adjusted_cost_factor = 1`.
- Clamp invalid or negative parsed values to zero before computing.

Nominal price score is normalized within a cohort of comparable managed channels:

```text
cohort = local_group + channel_type
```

Within a cohort, a lower nominal source/channel rate gets a higher `nominal_price_score`; cache factor is excluded from the floor, bounds, and score. The backward-compatible `effective_price_score` field aliases this nominal score. `effective_cost_multiplier` remains an explanatory diagnostic only.

When a cohort has only one managed channel with a valid nominal rate, assign `nominal_price_score = 100`.

The guarded cache factor maps independently and monotonically to `cache_score`: `1.0` or worse maps to `0`, the existing `0.35` floor maps to `100`, and intermediate values are linearly interpolated. A no-sample channel scores exactly `95`, using the inverse-mapped factor `0.3825`, regardless of same-cohort peer data. Counts 1–19 blend that prior toward the channel's own bounded factor with confidence `usage_log_count / 20`; at 20 or more samples, the own factor is used completely.

### Availability

Availability comes from channel monitor history over the resolved metrics window, capped at 7 days because monitor retention is 7 days.

```text
availability = successful_checks / total_checks
```

Statuses counted as successful:

- `success`
- `degraded`

Score:

```text
availability_score = availability * 100
```

If there is no monitor data:

```text
availability_score = 70
```

The neutral default avoids over-penalizing newly synced channels while still letting monitored reliable channels outrank unknown channels when prices are close.

### First Token Latency

First-token latency should use real request logs, not monitor probes, for the primary score. The field is `other.frt`.

Only positive `frt` values from real requests are used. Manual channel tests are excluded.

Normalize latency within the same cohort as price. Lower latency gets a higher score. Since latency has only 3% weight, missing data should not materially change routing:

```text
first_token_score = 70
```

when no real first-token samples exist.

## Hysteresis and Stability

Automatic priority must avoid noisy priority churn.

The first version should use these stability rules:

- Only write priority when `abs(new_priority - old_priority) >= 10`.
- Always write when no previous automatic-priority snapshot exists and the feature is enabled.
- Do not update more often than the resolved interval unless interval is `0`.
- If a channel no longer has a valid upstream mapping or effective multiplier, skip it and record a score error in the snapshot.
- Manual priority edits after automatic priority is enabled may be overwritten by the next run. The UI should make this ownership clear.

A due run should update the score snapshot even when hysteresis prevents changing `priority`. In that case:

- `Applied` is `false`;
- `Reason` is `hysteresis_delta_below_threshold`;
- `OldPriority` and `NewPriority` record the current priority and computed priority;
- `channels.priority` and `abilities.priority` are not changed.

This gives the UI an up-to-date explanation without causing scheduler churn.

## Persistence

Store the latest resolved automatic-priority state and score explanation in `dto.ChannelOtherSettings`.

Suggested fields:

```go
ChannelAutoPriorityEnabled bool
ChannelAutoPriorityIntervalMinutes int
ChannelAutoPriorityWindowHours int
ChannelAutoPriorityLastRunAt int64
ChannelAutoPriorityLastAppliedAt int64
ChannelAutoPriorityLastScore *ChannelAutoPriorityScore
```

Suggested score payload:

```go
type ChannelAutoPriorityScore struct {
    Version string
    ComputedAt int64
    WindowStart int64
    WindowEnd int64
    Cohort string
    EffectiveRateMultiplier float64
    CacheAdjustedCostFactor float64
    EffectiveCostMultiplier float64
    EffectivePriceScore float64
    AvailabilityScore float64
    FirstTokenScore float64
    FinalScore float64
    OldPriority int64
    NewPriority int64
    Applied bool
    Reason string
    UsageLogCount int64
    MonitorCheckCount int64
    FirstTokenSampleCount int64
}
```

Do not add a full history table in the first implementation. The latest snapshot is enough for UI explanation and avoids extra migration risk. A history table can be added later if the dashboard needs trend charts.

## Worker Design

Create an upstream-source auto-priority worker parallel to upstream-source auto-sync:

- master node only;
- wakes every minute;
- protected by an atomic running flag;
- lists enabled upstream sources with any automatic-priority schedule;
- loads mappings for each source;
- resolves rule-level automatic-priority settings for each mapping;
- filters to due mappings with `local_channel_id > 0`;
- loads channels and verifies generated ownership metadata matches source and mapping;
- computes all channel stats in batches;
- writes changed priorities and score snapshots.

The worker should process a source in a bounded context timeout. It should log source-level failures and continue with other sources.

Manual API endpoint:

```text
POST /api/upstream-sources/:id/auto-priority/run
```

This endpoint runs the same scorer for one source immediately and returns per-channel results. It is useful for testing the formula before waiting for the scheduler.

## Database and Cache Updates

When a priority is applied:

1. Update `channels.priority`.
2. Update `channels.settings` with the latest auto-priority snapshot.
3. Update `abilities.priority` for the channel.
4. Refresh memory channel cache after a batch that changed at least one priority.

The implementation should use GORM APIs and avoid JSON SQL extraction so SQLite, MySQL, and PostgreSQL stay compatible.

Real usage stats should be loaded by indexed columns first:

```text
type = consume
channel_id IN (...)
created_at >= window_start
```

Then parse `Log.Other` in Go through `common.UnmarshalJsonStr` or existing wrappers.

## UI Design

Upstream source edit dialog:

- Add an "Automatic Priority" section near sync/monitor settings.
- Source-level controls:
  - enable automatic priority;
  - interval minutes;
  - metrics window hours.
- Local group rule controls:
  - enable override;
  - interval override;
  - metrics window override.

Channel table/detail:

- Show a compact automatic-priority indicator for managed generated channels.
- The indicator should show final score, nominal rate/price score, cache factor/cache score, availability, and last applied time. Effective cost may remain labeled as a diagnostic.
- Keep manual priority editing visible, but show that automatic priority will overwrite it when enabled.

All new frontend labels must use `t('English key')` and be added to all locale files through the i18n workflow.

## Error Handling

- Invalid source sync config: skip the source and log a warning.
- Missing mapping: skip the channel and write a snapshot reason when possible.
- Missing effective multiplier: skip priority write; snapshot reason `missing_effective_rate_multiplier`.
- Missing usage logs: use cache factor `1` and first-token score `70`.
- Missing monitor logs: use availability score `70`.
- Invalid JSON in log `other`: ignore that log for usage-derived metrics.
- Database update failure: return the failed channel result from the manual endpoint and log the worker failure.

## Testing Strategy

Backend tests should be deterministic and behavior-focused.

Required model/service tests:

1. Nominal price uses the source/channel rate only and is unchanged by cache behavior.
2. Cache-heavy usage raises the independent cache score when cache ratio is below normal input price.
3. Cache creation fields can increase or decrease the cache-adjusted cost according to their ratios.
4. Manual channel test logs are excluded from cache and first-token stats.
5. Missing usage data falls back to raw multiplier and neutral first-token score.
6. Availability uses monitor logs in the requested window and treats `success` and `degraded` as successful.
7. Score normalization compares only channels in the same `local_group + channel_type` cohort.
8. Priority write updates channel priority, ability priority, and score snapshot together.
9. Hysteresis prevents small priority changes from being written.
10. Only upstream-source-generated channels with matching source/mapping metadata are eligible.
11. Rule-level automatic-priority settings override source-level settings.
12. The due-source listing honors interval `0` as always due.
13. A due run with interval `0` still honors hysteresis and does not rewrite channel or ability priority when the computed delta is below threshold.
14. A hysteresis no-op still writes an explanatory score snapshot with `Applied=false`.

Frontend tests:

1. Upstream source form preserves source-level automatic-priority settings.
2. Rule editor preserves automatic-priority overrides.
3. Generated channel automatic-priority explanation renders translated labels.
4. Manual priority editor warns when automatic priority owns the generated channel.
