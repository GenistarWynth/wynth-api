# Auto Priority Cache Stability Design

## Goal

Automatic priority scoring should react to real cache economics without letting a small or unusually cache-heavy window dominate routing decisions. Cache behavior is dynamic, so the score must be useful when there is enough data and conservative when there is not.

## Scope

This change only affects channels generated by upstream source sync and only changes the cache-adjusted cost component of automatic priority scoring. The existing final-score weights remain unchanged:

- effective price score: 80%
- availability score: 17%
- first-token latency score: 3%

## Rules

1. Low sample guard:
   - fewer than 5 usage logs: cache factor is treated as `1.0`;
   - 5 to 19 usage logs: cache factor is blended toward `1.0`;
   - 20 or more usage logs: the measured cache factor is trusted.

2. Cache benefit floor:
   - cache-adjusted cost factor cannot go below `0.35`;
   - this prevents short bursts of extreme cache hits from making a channel appear nearly free.

3. Historical smoothing:
   - when a previous auto-priority snapshot has a valid effective cost multiplier, the current effective cost is smoothed:

```text
smoothed_effective_cost =
  0.65 * current_effective_cost
  + 0.35 * previous_effective_cost
```

4. Existing safeguards remain:
   - missing or invalid upstream effective rate skips priority updates;
   - same-cohort relative price scoring stays unchanged;
   - hysteresis still suppresses priority changes smaller than 10 points after the first snapshot.

## Stored Snapshot

The channel snapshot continues to store the cost factor and effective cost actually used for scoring. This keeps the channel table and future smoothing aligned with the applied decision.
